## Project Overview

Movie KGQA (Knowledge Graph Question Answering) system using Neo4j 5.x, BERT-base-chinese, FastAPI, and Streamlit. Natural language questions → intent recognition → entity linking → parameterized Cypher → graph query → answer generation.

## Commands

```bash
# Activate uv-managed venv (Python 3.8–3.10)
source .venv/bin/activate

# Install/refresh dependencies
uv pip install -r requirements.txt

# Run tests
uv run python -m unittest discover -s tests -p 'test*.py'

# Run Neo4j via Docker
docker run -d --name neo4j-kgqa -p 7474:7474 -p 7687:7687 \
    -e NEO4J_AUTH=neo4j/20040121 \
    -e NEO4J_dbms_memory_heap_max__size=1G \
    -v $(pwd)/neo4j_data:/data -v $(pwd):/var/lib/neo4j/import neo4j:5

# Data pipeline: preprocess CSV → import into Neo4j
uv run python data_tools/data_preprocess.py
uv run python data_tools/neo4j_import.py

# Train BERT intent model (optional; rule-based fallback works without it)
uv run python intent_model_train.py

# Start services (two terminals)
uv run uvicorn main_api:app --reload --host 0.0.0.0 --port 8000
uv run streamlit run app.py --server.port 8501
```

## Architecture

**Processing pipeline** (all in `core_modules.py`):
1. `IntentRecognizer` — BERT classifier with keyword rule fallback (threshold: 0.55). 5 intents: query_director, query_actor, query_rating, query_year, query_genre.
2. `EntityLinker` — Alias → exact match → difflib → BERT cosine similarity → intent-guided ranking. Pre-computes entity vectors at startup.
3. `CypherGenerator` — Intent + entity type → parameterized Cypher template. Never string-interpolate user input into queries.
4. `Neo4jSession` — Neo4j 5.x via `neo4j://` protocol. Credentials in `core_modules.py` top-level constants (NEO4J_URI, NEO4J_USER, NEO4J_PWD). `neo4j://localhost:7687` / `neo4j` / `20040121`.
5. `AnswerGenerator` — Template-based (zero latency) or DeepSeek LLM (via OpenAI-compatible API). Configured through env vars, not hardcoded keys.
6. `ConversationManager` — Pronoun resolution, entity memory (last 5 turns), suggested follow-up questions.

**Service layer** (`main_api.py`): FastAPI with 10 endpoints. Singleton instances of Neo4jSession, IntentRecognizer, EntityLinker, AnswerGenerator at module level. In-memory session store (`sessions` dict keyed by session_id). CORS open to all origins.

**Frontend** (`app.py`): Streamlit with chat-style UI, ECharts force-directed graph visualization, expandable reasoning trace per answer.

**Config** (`config.py`): Reads `LLM_API_KEY`, `LLM_URL`, `LLM_MODEL` from env or local `.env`. Respects `KGQA_LOAD_DOTENV=0` to skip dotenv. `LLMConfig` dataclass, `get_llm_config()`, `has_llm_api_key()`, `get_llm_status()` helpers.

## Important Constraints

- **Python 3.8–3.10 only** — PyTorch 1.13.1 compatibility.
- **Neo4j 5.x** — uses `neo4j://` protocol (not `bolt://`). Must be running before API starts.
- **`intent_model.pth`** — 400MB BERT weights, listed in `.gitignore`. Generated by `intent_model_train.py`.
- **`.env` is gitignored** — copy `.env.example` and fill in real keys. Default model is DeepSeek but any OpenAI-compatible endpoint works.
- **Flat module structure** — `core_modules.py` contains all 6 pipeline classes (~900 lines). `main_api.py` and `app.py` both import directly from it. No package hierarchy.
- **AGENTS.md** exists with additional guidelines on coding style, commit conventions, and security. Some paths in it are stale (scripts moved to `data_tools/` and `tests/` subdirectories).
